### Behavior 1: register user

Behavior name:
register user

Successful execution:
- Result:
  This behavior creates a disabled user account, stores a verification token, and sends an activation email.
- Endpoint sequence:
  Step 1: `POST /api/auth/register` with body fields `username`, `name`, `email`, `password`, and `confirmPassword`.
- Constraints:
  `username`, `name`, `email`, `password`, and `confirmPassword` must be nonblank. `email` must be email-shaped. `password` must be 8 to 20 characters. `username` must not already exist. Implementation stores `email`, `username`, and encoded `password`; `name` is not persisted and `confirmPassword` is not compared with `password`.

Failure or exceptional branches:
- Branch 1:
  - Unsatisfied condition:
    The username is already registered.
  - Endpoint group:
    Step 1: `POST /api/auth/register` with `username = X`.
    Step 2: `POST /api/auth/register` again with `username = X`.
  - Failure endpoint:
    `POST /api/auth/register`
  - Why this fails:
    The controller checks `existsByUsername` and returns HTTP 400 with message `Username already exists`.
  - Intentionally violated constraints:
    The second request reuses an existing `username`.
- Branch 2:
  - Unsatisfied condition:
    The request body violates DTO validation.
  - Endpoint group:
    Step 1: `POST /api/auth/register` with blank fields, invalid `email`, or `password` outside 8 to 20 characters.
  - Failure endpoint:
    `POST /api/auth/register`
  - Why this fails:
    The endpoint uses `@Valid` on `RegisterRequestDto`.
  - Intentionally violated constraints:
    Required fields or validation bounds are not satisfied.

Endpoint coverage:
- Covers:
  `POST /api/auth/register`
- Distinct meaning:
  Creates a disabled account and initiates email verification.
